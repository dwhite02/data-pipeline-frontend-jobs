What You’re Building & Why It Matters

This answers a real question people in tech are actively debating right now. Your pipeline surfaces data-driven insights on the shifting frontend hiring landscape.

01
Is the share of entry-level frontend roles underrepresented in current job postings?
Seniority distribution analysis across 2023–2024 LinkedIn postings. Cross-reference with Stack Overflow survey AI adoption data for developer sentiment context (GitHub Copilot mid-2022, ChatGPT late 2022).

Proxy: applicant count vs. listing age. Flag postings with 100+ applicants still listed after 60 days as potential ghost jobs.

0.4Verify & Set Up Local Tools

Step 1 — Python via pyenv (do this first)

macOS ships with an old system Python you should never modify. The cleanest, most reliable way to manage Python on a Mac is pyenv — it sets a true global default that persists across every terminal session without PATH conflicts.

⚠️ Why not python.org installer or Homebrew Python directly?
The python.org installer and Homebrew Python both work but require manually editing ~/.zshrc PATH entries that can get overridden by other tools. pyenv uses shell shims that always win — no PATH fights. It's the standard approach recommended by the pyenv docs and the broader macOS developer community.
💡 Which Python version to install?
Install Python 3.12. Per the official dbt docs, Python 3.13 has a warning flag (⚠) on the BigQuery adapter — not all dependencies support it yet. Python 3.12 is fully supported across dbt Core v1.8–v1.11 and all other tools in this project.
Terminal — install pyenv and Python 3.12 
# Step 1: Install Homebrew if you don't have it
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Step 2: Install pyenv via Homebrew (recommended method per pyenv docs)
brew install pyenv

# Step 3: Add pyenv to your shell — open ~/.zshrc and add these lines at the bottom
open ~/.zshrc

Add these three lines to the bottom of your ~/.zshrc file:

export PYENV_ROOT="$HOME/.pyenv"
command -v pyenv >/dev/null || export PATH="$PYENV_ROOT/bin:$PATH"
eval "$(pyenv init -)"
⚠️ The eval line is required — don't skip it
eval "$(pyenv init -)" activates pyenv's shell shims every time a terminal opens. Without this line pyenv installs Python fine but your terminal won't actually use it — it'll fall back to the system Python. This is the most common pyenv setup mistake.
Terminal — reload shell and install Python 3.12 
# Reload your shell config so pyenv activates immediately
source ~/.zshrc

# Install Python 3.12 (fully supported by all tools in this project)
pyenv install 3.12

# Set 3.12 as your global default for all terminals

# Open a NEW terminal tab and verify it stuck

pip3 --version      # should show pip 24.x or 25.x from python 3.12
✅ How to confirm pyenv is working correctly
Run pyenv versions — you should see an asterisk (*) next to 3.12 indicating it's the active global version. Run which python3 — it should show a path containing .pyenv/shims/. If both of those look right, you're set.

💡 Why Terraform instead of clicking in the GCP console?
You could create a GCS bucket by clicking around in the GCP UI. But that's not reproducible — someone cloning your repo would have to manually recreate every resource. terraform/main.tf 
# Tell Terraform we're using Google Cloud
terraform {
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 7.0"
    }
  }
}

# Configure the provider — credentials path comes from terraform.tfvars
provider "google" {
  credentials = file(var.credentials_path)
  project     = var.project_id
  region      = var.region
}

# GCS bucket = your DATA LAKE
# Raw files land here before being loaded into BigQuery
resource "google_storage_bucket" "data_lake" {
  name          = var.bucket_name
  location      = upper(var.region)
  force_destroy = true

  lifecycle_rule {
    condition { age = 30 }   # auto-delete raw files after 30 days (cost saving)
    action    { type = "Delete" }
  }
  uniform_bucket_level_access = true
}

# BigQuery dataset = your DATA WAREHOUSE
# Transformed, queryable tables live here
resource "google_bigquery_dataset" "warehouse" {
  dataset_id                 = var.bq_dataset_id
  location                   = var.region
  delete_contents_on_destroy = true
}

# Output what was created so you can see it after terraform apply
output "gcs_bucket_name" { value = google_storage_bucket.data_lake.name }
output "bq_dataset"      { value = google_bigquery_dataset.warehouse.dataset_id }

2.2Configure GCP Credentials in Kestra

Same setup as Module 2 — KV pairs for config values, a Secret for the credential JSON.

Step 1: In the Kestra UI, go to Namespaces and create a namespace called capstone. Then create and execute this flow:

kestra/flows/01_setup_gcp_kv.yaml

namespace: capstone
tasks:

    type: io.kestra.plugin.core.kv.Set
    key: GCP_PROJECT_ID
    kvType: STRING
    value: YOUR-PROJECT-ID   # replace
  - id: location

Paste this into a new flow in the Kestra UI (Flows → Create), save, then click Execute.

Step 2: Store your GCP credentials as a Kestra Secret. Per the official Kestra docs, the correct approach for Docker Compose is to base64-encode the JSON and store it in a .env_encoded file — Kestra reads this at startup and securely injects the secrets.

Terminal — run from your project root (not inside kestra/) 
# macOS uses -b 0 (not -w 0) to disable line wrapping on base64
echo "SECRET_GCP_CREDS=$(cat ~/.gcp/fe-jobs-sa.json | base64 -b 0)" >> kestra/.env_encoded

# Add your Kaggle token (replace with your actual token value)
echo "SECRET_KAGGLE_API_TOKEN=$(echo -n 'your_actual_token_here' | base64)" >> kestra/.env_encoded

# Verify the file was created
ls -la kestra/.env_encoded
⚠️ If you get "No such file or directory"
You are probably running the command from inside the kestra/ folder. The path kestra/.env_encoded only works from the project root. Either cd .. to go back to the root, or drop the kestra/ prefix and just use .env_encoded if you are already inside kestra/.
⚠️ Replace your_kaggle_token_here with your actual token
Your Kaggle API token is the value from kaggle.com/settings → API Tokens. The echo -n flag prevents a trailing newline from being included in the encoded value, which would corrupt the secret.

Start Kestra:

💡 Why base64 and not pasting JSON directly?
Kestra identifies secrets by scanning for environment variables prefixed with SECRET_ and base64-decodes their values at runtime. This keeps raw credentials out of plain-text environment variables and process logs. The SECRET_ prefix is stripped when you reference secrets in flows — so SECRET_GCP_CREDS is referenced as {{secret('GCP_CREDS')}} and SECRET_KAGGLE_API_TOKEN as {{secret('KAGGLE_API_TOKEN')}}.
💡 KV Store vs Secrets — what's the difference?
KV pairs are plain config values like project ID and bucket name. Fine to store in plain text, not sensitive.

Secrets are encrypted. Your service account JSON contains a private key — Kestra encrypts it and never exposes it in logs. In your flows you reference it as {{secret('GCP_CREDS')}} and Kestra decrypts it at runtime. Identical to what you did in Module 2 with 06_gcp_kv.yaml and 07_gcp_setup.yaml.

Data Warehouse — BigQuery Verification

Verify your raw tables are correctly structured and understand the data before writing dbt models. Partition + cluster live on the mart tables you build in Phase 4, which earns full Data Warehouse rubric points (4/4).
⏳ Estimated time: 30–45 min
💡 Why BigQuery specifically, not just Postgres?
BigQuery is an OLAP (Online Analytical Processing) columnar database built for analytics. When you ask "what's the average applicant count for entry-level jobs?", BigQuery only reads the two relevant columns across 124K rows instead of the entire table. Postgres (OLTP) reads entire rows. At this data scale, that difference translates to seconds vs minutes and pennies vs dollars in query costs. This is why data warehouses exist separately from operational databases.
⚠️ Watch your bytes — free tier is real but not infinite
Every query you run in BigQuery shows Bytes processed in the bottom panel of the console. This is your cost signal. At this project’s scale you’re comfortably inside the 1TB/month free tier — but SELECT * on repeated queries burns through it faster than necessary. During development: SELECT specific_columns LIMIT 100 instead of SELECT *.

External tables (raw SO survey) scan the full CSV on every query — there is no free partition-based pruning on external tables. Once your dbt staging model runs and materializes stg_stackoverflow_survey as a view, queries go through the mart table instead, which is more efficient.

A view is a saved SQL query — no stored data, runs fresh every call. A table stores computed results. -- Cleans and standardizes LinkedIn 2024 raw data.
-- Key jobs done here:
--   1. Rename columns to consistent names used across all models
--   2. Parse UNIX millisecond timestamps to real DATE type
--   3. Flag frontend roles by keyword-matching job titles
--   4. Normalize seniority labels (LinkedIn uses varied text like "Entry level", "Mid-Senior level")

WITH source AS (
  SELECT * FROM {{ source('frontend_jobs', 'raw_linkedin_job_postings') }}
),

cleaned AS (
  SELECT
    job_id,
    company_id,
    company_name,
    title                         AS job_title,
    formatted_experience_level    AS seniority_level,
    formatted_work_type           AS work_type,
    remote_allowed                AS is_remote,
    location,
    CAST(applies AS INT64)        AS applicant_count,
    CAST(views AS INT64)          AS listing_views,
    description                   AS job_description,

    -- original_listed_time is UNIX milliseconds as a float (e.g. 1713397508000.0)
    -- Must CAST to INT64 first before TIMESTAMP_MILLIS conversion
    DATE(TIMESTAMP_MILLIS(CAST(CAST(original_listed_time AS FLOAT64) AS INT64))) AS posted_date,
    DATE(TIMESTAMP_MILLIS(CAST(CAST(expiry AS FLOAT64) AS INT64)))               AS expiry_date,

    'linkedin_2023_2024'          AS data_source,
    EXTRACT(YEAR FROM DATE(TIMESTAMP_MILLIS(CAST(CAST(original_listed_time AS FLOAT64) AS INT64)))) AS post_year,

    -- Frontend role detection: TRUE if job title matches frontend keywords
    -- Adjust this list after previewing actual data titles from Phase 3
    CASE WHEN LOWER(title) LIKE ANY (
      '%frontend%', '%front-end%', '%front end%',
      '%react%', '%vue%', '%angular%',
      '%ui engineer%', '%ui developer%',
      '%web developer%', '%javascript developer%',
      '%next.js%', '%typescript developer%'
    ) THEN TRUE ELSE FALSE END    AS is_frontend_role,

    -- Normalize seniority labels to 3 clean categories
    -- LinkedIn uses labels like "Entry level", "Mid-Senior level", "Director" etc.
    CASE
      WHEN LOWER(formatted_experience_level) LIKE '%entry%'      THEN 'Entry Level'
      WHEN LOWER(formatted_experience_level) LIKE '%associate%'  THEN 'Entry Level'
      WHEN LOWER(formatted_experience_level) LIKE '%internship%' THEN 'Entry Level'
      WHEN LOWER(formatted_experience_level) LIKE '%mid%'        THEN 'Mid Level'
      WHEN LOWER(formatted_experience_level) LIKE '%senior%'     THEN 'Senior Level'
      WHEN LOWER(formatted_experience_level) LIKE '%director%'   THEN 'Senior Level'
      WHEN LOWER(formatted_experience_level) LIKE '%executive%'  THEN 'Senior Level'
      WHEN LOWER(formatted_experience_level) LIKE '%lead%'       THEN 'Senior Level'
      ELSE 'Unknown'
    END                           AS seniority_category

  FROM source
  WHERE title IS NOT NULL
    AND formatted_experience_level IS NOT NULL

💡 What is _FILE_NAME and why does it work here?
_FILE_NAME is a BigQuery pseudo-column available on all external tables. It contains the full GCS URI of the file that contributed each row — for example: gs://fe-jobs-lake-dw/raw/stackoverflow_survey/survey_2023.csv. It’s not a column in the CSV itself; BigQuery injects it automatically.

The REGEXP_EXTRACT(_FILE_NAME, r'survey_(\d{4})\.csv') pulls the 4-digit year from that path, giving every row a survey_year column even though the CSV files don’t contain a year column. This is the standard pattern for partitioned external table ingestion. BigQuery external table pseudo-columns docs →
⚠️ SO survey schema shifts across years — verify before dbt run
When BigQuery creates a wildcard external table from survey_*.csv, it auto-detects schema by sampling all three files. If columns present in 2023/2024 (like AISelect, AISent, AIToolCurrently Using) are absent in 2022, BigQuery may omit them from the schema entirely — causing dbt run on stg_stackoverflow_survey to error with "column not found".

If that happens: run the CREATE OR REPLACE EXTERNAL TABLE SQL from flow 04 manually in BigQuery Console (without skip_leading_rows) and inspect the inferred schema. Wrap any missing AI columns in IF('column_name' IN UNNEST(column_names), column_name, NULL) or use SAFE.column_name syntax. 2022 rows will naturally have NULL for AI fields.

Intermediate Layer — Unify the Two Job Datasets

models/intermediate/int_jobs_unified.sql 
-- Enriches LinkedIn job postings with skill counts from stg_job_skills.
-- This is the single source of truth for all mart models.
-- Industries join is an optional extension (raw_job_industries is available if needed).

WITH jobs AS (
  SELECT * FROM {{ ref('stg_linkedin_jobs') }}
),

skills AS (
  SELECT
    job_id,
    COUNT(*)          AS skill_count
  FROM {{ ref('stg_job_skills') }}
  GROUP BY 1
)

SELECT
  j.*,
  COALESCE(s.skill_count, 0) AS skill_count,

  -- Listing duration: days the job was listed before expiry.
  -- Uses expiry_date - posted_date so ghost detection works on a historical snapshot.
  -- CURRENT_DATE() would make every 2023-2024 posting appear 700+ days old.
  -- Rows without expiry_date will have NULL here (filtered in mart_ghost_postings).
  DATE_DIFF(j.expiry_date, j.posted_date, DAY) AS days_listed

FROM jobs j
LEFT JOIN skills s USING (job_id)
WHERE j.posted_date IS NOT NULL
💡 Why {{ ref() }} instead of a table name?
{{ ref('stg_linkedin_jobs') }} is dbt’s way of referencing another model. dbt resolves it to the real BigQuery table name AND builds a dependency graph — it automatically knows to run stg_linkedin_jobs before int_jobs_unified. If you hard-coded a table name like `project.dataset.stg_linkedin_jobs`, dbt can’t see that dependency and might run models in the wrong order. The ref() function is what makes dbt pipelines reliable and self-documenting.

models/marts/mart_seniority_trends.sql — answers Question 1 
-- QUESTION 1: What is the seniority distribution in frontend job postings?
-- Aggregates frontend job postings by year and seniority category.
-- Calculates pct_of_total so you can see relative seniority distribution.
--
-- No partition — this is a small aggregated table (a few hundred rows max).
-- Per Google BigQuery docs, tables under 64MB see negligible benefit from partitioning.
-- Cluster by seniority_category: queries GROUP BY / filter on this in every Looker tile.
-- Materialized as table — marts are always tables, never views.

-- dbt config: partition + cluster defined HERE in the mart, not in raw ingestion.
{{- config(
    materialized='table',
    cluster_by=['seniority_category', 'data_source']
) -}}

WITH jobs AS (
  SELECT * FROM {{ ref('int_jobs_unified') }}
  WHERE is_frontend_role = TRUE
    AND seniority_category != 'Unknown'
),

yearly_totals AS (
  SELECT post_year, COUNT(*) AS total_frontend_jobs
  FROM jobs GROUP BY 1
),

by_seniority AS (
  SELECT post_year, seniority_category, data_source, COUNT(*) AS job_count
  FROM jobs GROUP BY 1, 2, 3
)

SELECT
  s.post_year,
  s.seniority_category,
  s.data_source,
  s.job_count,
  t.total_frontend_jobs,
  ROUND(s.job_count / t.total_frontend_jobs * 100, 2) AS pct_of_total
FROM by_seniority s
JOIN yearly_totals t USING (post_year)
ORDER BY post_year, seniority_category
models/marts/mart_experience_inflation.sql — answers Question 2 
-- QUESTION 2: Are entry-level postings requiring senior-level skills?
-- "Experience inflation" = entry-level job title + demanding skill requirements.
-- Uses the LinkedIn skills table joined back to postings by job_id.
--
-- Partition by posted_date: date range filters are common in analysis queries.
-- Cluster by seniority_category, is_frontend_role: almost all filters use these.
-- Materialized as table — marts are always tables, never views.

-- dbt config: partition + cluster defined HERE in the mart, not in raw ingestion.
{{- config(
    materialized='table',
    partition_by={
      'field': 'posted_date',
      'data_type': 'date'
    },
    cluster_by=['company_name', 'location']
) -}}

WITH entry_jobs AS (
  SELECT * FROM {{ ref('stg_linkedin_jobs') }}
  WHERE is_frontend_role = TRUE
    AND seniority_category = 'Entry Level'
),

job_skills AS (
  SELECT * FROM {{ ref('stg_job_skills') }}
),

skills_analysis AS (
  SELECT
    j.job_id,
    j.job_title,
    j.company_name,
    j.posted_date,
    j.applicant_count,
    j.location,
    COUNT(s.skill_id)            AS total_skill_count,

    -- Count skills that signal senior/complex work in an entry-level posting
    COUNTIF(LOWER(s.skill_id) IN (
      'system design', 'aws', 'gcp', 'azure', 'kubernetes',
      'docker', 'ci/cd', 'microservices', 'architecture',
      'leadership', 'terraform', 'kafka'
    ))                           AS senior_skill_count,

    -- Inflation score: what % of required skills are senior-level?
    -- High score = entry job that expects senior knowledge
    ROUND(
      COUNTIF(LOWER(s.skill_id) IN (
        'system design', 'aws', 'gcp', 'azure', 'kubernetes',
        'docker', 'ci/cd', 'microservices', 'architecture',
        'leadership', 'terraform', 'kafka'
      )) / NULLIF(COUNT(s.skill_id), 0) * 100, 2
    )                            AS inflation_score

  FROM entry_jobs j
  LEFT JOIN job_skills s USING (job_id)
  GROUP BY 1,2,3,4,5,6
)

SELECT * FROM skills_analysis
ORDER BY inflation_score DESC NULLS LAST
⚠️ Validate skill_abr values before running this mart
The COUNTIF(LOWER(s.skill_id) IN ('aws', 'kubernetes', ...)) list assumes skill_abr values are readable lowercase strings. In reality, they can be short codes or mixed-case (e.g., JAVA, AWSEC2, LNKDNADS). If the list doesn’t match the actual values, every row will show senior_skill_count = 0 and the Q2 analysis will be meaningless even though the model runs without errors.

Before finalizing this mart: run this in BigQuery first: 
SELECT DISTINCT skill_id, COUNT(*) AS freq
FROM `YOUR_PROJECT.frontend_jobs.stg_job_skills`
GROUP BY 1
ORDER BY freq DESC
LIMIT 50;
Review the top 50 values and update the IN (...) list to match the codes that actually appear. This is standard exploratory analysis before writing mart logic.
models/marts/mart_ghost_postings.sql — answers Question 3 
-- QUESTION 3: Which companies show ghost posting patterns?
-- Ghost posting proxy: high applicant count + long listing duration.
-- Threshold: 100+ applicants AND listed for 60+ days (expiry_date - posted_date) = potential ghost job.
-- Uses listing_duration from the dataset itself, not CURRENT_DATE(), since this is a
-- historical snapshot. Rows without expiry_date are excluded (days_listed IS NOT NULL).
--
-- No partition — this is a small company-level aggregation table.
-- Cluster by company_name: Looker bar chart and filters operate on company.
-- Materialized as table — marts are always tables, never views.

-- dbt config: partition + cluster defined HERE in the mart, not in raw ingestion.
{{- config(
    materialized='table',
    cluster_by=['company_name']
) -}}

WITH linkedin_frontend AS (
  SELECT * FROM {{ ref('int_jobs_unified') }}
  WHERE data_source = 'linkedin_2023_2024'
    AND is_frontend_role = TRUE
    AND applicant_count IS NOT NULL
    AND days_listed IS NOT NULL
),

flagged AS (
  SELECT
    job_id,
    job_title,
    company_name,
    seniority_category,
    applicant_count,
    posted_date,
    days_listed,
    location,
    is_remote,

    CASE
      WHEN applicant_count >= 100
       AND days_listed >= 60 THEN TRUE
      ELSE FALSE
    END AS is_potential_ghost

  FROM linkedin_frontend
),

-- Aggregate to company level for the dashboard bar chart
company_summary AS (
  SELECT
    company_name,
    COUNT(*)                          AS total_frontend_postings,
    COUNTIF(is_potential_ghost)       AS ghost_posting_count,
    ROUND(COUNTIF(is_potential_ghost) / NULLIF(COUNT(*), 0) * 100, 1) AS ghost_pct,
    AVG(applicant_count)              AS avg_applicants,
    AVG(days_listed)                  AS avg_days_listed
  FROM flagged
  GROUP BY 1
  HAVING total_frontend_postings >= 3   -- exclude companies with only 1-2 posts
)

SELECT * FROM company_summary
ORDER BY ghost_pct DESC
💡 Ghost postings — understanding the proxy and its limits
This mart uses a proxy definition: 100+ applicants AND listed for 60+ days before expiry. It identifies suspicious patterns, not proven bad intent. A legitimate job might have 100 applicants and stay open for months during a slow hiring process.

Two important limitations to document in your README and dashboard:
1. NULL bias: The applicant_count IS NOT NULL and days_listed IS NOT NULL filters exclude all postings where LinkedIn didn’t surface those values. You’re only analyzing the subset where both were visible — which skews toward more popular or prominently listed jobs.
2. Survivorship bias: Only postings that were active long enough to accumulate applicants appear here. Short-lived legitimate postings are excluded.

Acknowledging these limitations in your dashboard actually strengthens your project — it shows analytical maturity that peer reviewers and employers value.

💡 Why tests matter — they protect your analysis
If job_id is not unique in the source data, the LEFT JOIN in int_jobs_unified silently multiplies rows — your applicant averages become inflated and the ghost posting mart shows phantom duplicates. A unique test on job_id catches this before it reaches Looker and corrupts your charts. Tests are not bureaucracy — they’re the circuit breaker between bad source data and wrong conclusions.

The rule applied to this project
Raw layer (Kestra → BigQuery): flat CREATE OR REPLACE TABLE AS SELECT * — no partition, no cluster. Fast writes, simple schema.

Mart layer (dbt): each mart has a {{ config() }} block with cluster (and partition where warranted) based on actual Looker query patterns:
  • mart_seniority_trends — cluster only: seniority_category, data_source (small table, <64MB — partitioning adds no benefit per Google docs)
  • mart_experience_inflation — partition by posted_date, cluster by seniority_category, is_frontend_role
  • mart_ghost_postings — cluster only: company_name (small aggregated table)
