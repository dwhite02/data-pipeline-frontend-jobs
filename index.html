What You’re Building & Why It Matters

This answers a real question people in tech are actively debating right now. Your pipeline surfaces data-driven insights on the shifting frontend hiring landscape.

01
Has the share of entry-level frontend roles declined since 2019, and when did the shift happen?
Temporal trend across 2019–2024. Correlate the inflection point with AI tool adoption (Copilot launched 2022, ChatGPT late 2022).
02
Are entry-level frontend postings requiring mid/senior-level skills — “experience inflation”?
Cross-reference job title seniority with required skills. Surface the mismatch between what a role is called and what it actually demands.
03
Which companies show ghost posting patterns — high applicants, long-listed, never filled?
Proxy: applicant count vs. listing age. Flag postings with 100+ applicants still listed after 60 days as potential ghost jobs.
📌 Problem Statement (copy this into your README)
The frontend engineering job market has experienced significant shifts in recent years, with debates around AI impact and a perceived decline in entry-level opportunities. This project builds an end-to-end batch data pipeline analyzing job postings from 2019–2024 — examining seniority trends, experience inflation, and ghost posting patterns — to surface data-driven insights into who’s actually hiring frontend engineers and what they actually expect.

💡 What's a dbt adapter?
dbt Core itself is database-agnostic — it doesn't know how to talk to BigQuery, Snowflake, or Postgres on its own. An "adapter" is a plugin that teaches dbt how to connect to a specific database. Since your warehouse is BigQuery, you install dbt-bigquery. This is why dbt is so widely used in industry: the same transformation framework works across any cloud warehouse.

Phase 1

Infrastructure as Code (Terraform)

Provision your GCS data lake bucket and BigQuery dataset using Terraform. This earns the Cloud + IaC rubric category (4/4 points).
⏳ Estimated time: 45 min – 1.5 hours
💡 Why Terraform instead of clicking in the GCP console?
You could create a GCS bucket by clicking around in the GCP UI. But that's not reproducible — someone cloning your repo would have to manually recreate every resource. Terraform lets you define infrastructure as code: one terraform apply command and everything is created identically every time. This is how real engineering teams manage cloud infrastructure at scale, and it's a specific rubric requirement worth 4 points.
1.1Create a GCP Service Account

A service account is like a robot user in GCP — it has its own credentials and permissions. This is the same concept you used in Module 2 when Kestra needed {{secret('GCP_CREDS')}} to talk to GCP.

  1. Go to GCP Console → IAM & Admin → Service Accounts
  2. Click Create Service Account
  3. Name: fe-jobs-sa → Click Create and Continue
  4. Add these roles (one at a time using Add Role):
    • Storage Admin — create/manage GCS buckets
    • BigQuery Admin — create/manage datasets and tables
    • Viewer — baseline read access
  5. Click Continue → Done
  6. Click on the service account → Keys tab → Add Key → Create new key → JSON
  7. Save the downloaded file securely outside your repo
Terminal — move and secure the key 
mkdir -p ~/.gcp
mv ~/Downloads/your-key-file.json ~/.gcp/fe-jobs-sa.json
chmod 600 ~/.gcp/fe-jobs-sa.json   # only you can read this file
⚠️ Treat this file like a password
The JSON file contains a private key that grants full access to your GCP project. Do not put it in your repo, do not paste it in Slack or Discord, do not email it. You'll reference it by its file path in Terraform and by its contents as a secret in Kestra. The .gitignore from Phase 0 already blocks *.json.
1.2Write Your Terraform Configuration

Create three files inside your terraform/ folder.

terraform/variables.tf 
# Declaring variables here keeps main.tf clean and generic.
# Anyone can reuse your infrastructure code with their own values.
variable "project_id" {
  description = "Your GCP Project ID"
  type        = string
}
variable "region" {
  description = "GCP region"
  type        = string
  default     = "us-central1"
}
variable "bucket_name" {
  description = "GCS bucket name — must be globally unique across all of GCP"
  type        = string
}
variable "bq_dataset_id" {
  description = "BigQuery dataset name"
  type        = string
  default     = "frontend_jobs"
}
terraform/main.tf 
# Tell Terraform we're using Google Cloud
terraform {
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
  }
}

# Configure the provider with your credentials
provider "google" {
  credentials = file("~/.gcp/fe-jobs-sa.json")
  project     = var.project_id
  region      = var.region
}

# GCS bucket = your DATA LAKE
# Raw files land here before being loaded into BigQuery
resource "google_storage_bucket" "data_lake" {
  name          = var.bucket_name
  location      = upper(var.region)
  force_destroy = true

  lifecycle_rule {
    condition { age = 30 }   # auto-delete raw files after 30 days (cost saving)
    action    { type = "Delete" }
  }
  uniform_bucket_level_access = true
}

# BigQuery dataset = your DATA WAREHOUSE
# Transformed, queryable tables live here
resource "google_bigquery_dataset" "warehouse" {
  dataset_id                 = var.bq_dataset_id
  location                   = var.region
  delete_contents_on_destroy = true
}

# Output what was created so you can see it after terraform apply
output "gcs_bucket_name" { value = google_storage_bucket.data_lake.name }
output "bq_dataset"      { value = google_bigquery_dataset.warehouse.dataset_id }
terraform/terraform.tfvars (GITIGNORED — your actual values) 
project_id    = "YOUR-GCP-PROJECT-ID"     # replace with your actual project ID
region        = "us-central1"
bucket_name   = "fe-jobs-lake-yourname"  # must be globally unique — add your name/initials
bq_dataset_id = "frontend_jobs"
⚠️ Also create a terraform.tfvars.example
Since terraform.tfvars is gitignored, create a terraform.tfvars.example file with placeholder values and commit that instead. This shows reviewers the structure without exposing your real project ID: 
project_id    = "your-gcp-project-id"

💡 What is Kestra actually doing here?
Kestra is your orchestrator — it coordinates each step of your pipeline in the correct order, handles failures gracefully, provides a UI to monitor runs, and makes the whole thing schedulable. Without orchestration you'd manually run scripts one by one. In production, companies use Kestra, Airflow, or Prefect to manage hundreds of these pipelines running on schedules. By using Kestra you're demonstrating industry-relevant skills beyond just writing scripts.
2.2Configure GCP Credentials in Kestra

Same setup as Module 2 — KV pairs for config values, a Secret for the credential JSON.

Step 1: In the Kestra UI, go to Namespaces and create a namespace called capstone. Then create and execute this flow:

kestra/flows/01_setup_gcp_kv.yaml 
id: 01_setup_gcp_kv
namespace: capstone
tasks:
  - id: project_id
    type: io.kestra.plugin.core.kv.Set
    key: GCP_PROJECT_ID
    kvType: STRING
    value: YOUR-PROJECT-ID   # replace
  - id: location
    type: io.kestra.plugin.core.kv.Set
    key: GCP_LOCATION
    kvType: STRING
    value: US-CENTRAL1
  - id: bucket
    type: io.kestra.plugin.core.kv.Set
    key: GCP_BUCKET_NAME
    kvType: STRING
    value: fe-jobs-lake-yourname  # must match your Terraform bucket name exactly
  - id: dataset
    type: io.kestra.plugin.core.kv.Set
    key: GCP_DATASET
    kvType: STRING
    value: frontend_jobs

Paste this into a new flow in the Kestra UI (Flows → Create), save, then click Execute.

Step 2: Store your GCP credentials as a Kestra Secret. Go to Namespaces → capstone → KV Store → Secrets → New Secret:

  • Key: GCP_CREDS
  • Value: paste the entire contents of ~/.gcp/fe-jobs-sa.json

Step 3: Store your Kaggle credentials as Secrets too:

  • Key: KAGGLE_USERNAME → your Kaggle username (from kaggle.json)
  • Key: KAGGLE_KEY → your Kaggle API key (from kaggle.json)
💡 KV Store vs Secrets — what's the difference?
KV pairs are plain config values like project ID and bucket name. Fine to store in plain text, not sensitive.

Secrets are encrypted. Your service account JSON contains a private key — Kestra encrypts it and never exposes it in logs. In your flows you reference it as {{secret('GCP_CREDS')}} and Kestra decrypts it at runtime. Identical to what you did in Module 2 with 06_gcp_kv.yaml and 07_gcp_setup.yaml.
2.3Build the Ingestion Pipeline Flows

Three flows, one per dataset. Each follows: download → upload to GCS → load to BigQuery.

kestra/flows/02_ingest_linkedin_2024.yaml 
id: 02_ingest_linkedin_2024
namespace: capstone
description: Downloads 1.3M LinkedIn job postings from Kaggle, uploads to GCS data lake, loads into BigQuery raw table with partitioning and clustering.

variables:
  dataset_slug: "asaniczka/1-3m-linkedin-jobs-and-skills-2024"
  gcs_prefix: "gs://{{kv('GCP_BUCKET_NAME')}}/raw/linkedin_2024/"

tasks:
  # STEP 1: Download from Kaggle. Kaggle CLI unzips automatically.
  - id: download_dataset
    type: io.kestra.plugin.scripts.shell.Commands
    outputFiles:
      - "*.csv"
    taskRunner:
      type: io.kestra.plugin.core.runner.Process
    commands:
      - pip install kaggle -q
      - export KAGGLE_USERNAME={{secret('KAGGLE_USERNAME')}}
      - export KAGGLE_KEY={{secret('KAGGLE_KEY')}}
      - kaggle datasets download -d {{vars.dataset_slug}} --unzip .
      - ls -la *.csv   # log what files were downloaded

  # STEP 2: Upload to GCS (your data lake). Same plugin as Module 2.
  - id: upload_jobs_to_gcs
    type: io.kestra.plugin.gcp.gcs.Upload
    from: "{{outputs.download_dataset.outputFiles['job_postings.csv']}}"
    to: "{{render(vars.gcs_prefix)}}job_postings.csv"

  - id: upload_skills_to_gcs
    type: io.kestra.plugin.gcp.gcs.Upload
    from: "{{outputs.download_dataset.outputFiles['job_skills.csv']}}"
    to: "{{render(vars.gcs_prefix)}}job_skills.csv"

  # STEP 3: Create external table pointing at GCS file
  - id: create_external_table
    type: io.kestra.plugin.gcp.bigquery.Query
    sql: |
      CREATE OR REPLACE EXTERNAL TABLE
        `{{kv('GCP_PROJECT_ID')}}.{{kv('GCP_DATASET')}}.linkedin_jobs_2024_ext`
      OPTIONS (
        format = 'CSV',
        uris = ['{{render(vars.gcs_prefix)}}job_postings.csv'],
        skip_leading_rows = 1,
        ignore_unknown_values = TRUE
      );

  # STEP 4: Materialize into a real table with PARTITION + CLUSTER
  # Partition by date = BigQuery only scans the dates you query (fast + cheap)
  # Cluster by seniority = rows sorted so seniority filters skip irrelevant data
  - id: load_partitioned_table
    type: io.kestra.plugin.gcp.bigquery.Query
    sql: |
      CREATE OR REPLACE TABLE
        `{{kv('GCP_PROJECT_ID')}}.{{kv('GCP_DATASET')}}.raw_linkedin_jobs_2024`
      PARTITION BY DATE(original_listed_time)
      CLUSTER BY formatted_experience_level, formatted_work_type
      AS SELECT * FROM
        `{{kv('GCP_PROJECT_ID')}}.{{kv('GCP_DATASET')}}.linkedin_jobs_2024_ext`;

  - id: load_skills_table
    type: io.kestra.plugin.gcp.bigquery.Query
    sql: |
      CREATE OR REPLACE EXTERNAL TABLE
        `{{kv('GCP_PROJECT_ID')}}.{{kv('GCP_DATASET')}}.raw_linkedin_skills`
      OPTIONS (
        format = 'CSV',
        uris = ['{{render(vars.gcs_prefix)}}job_skills.csv'],
        skip_leading_rows = 1,
        ignore_unknown_values = TRUE
      );

  - id: purge_files
    type: io.kestra.plugin.core.storage.PurgeCurrentExecutionFiles

# Injects GCP credentials into ALL gcp.* plugin tasks automatically
# Exact same pattern as your 08_gcp_taxi.yaml from Module 2
pluginDefaults:
  - type: io.kestra.plugin.gcp
    values:
      serviceAccount: "{{secret('GCP_CREDS')}}"
      projectId: "{{kv('GCP_PROJECT_ID')}}"
      location: "{{kv('GCP_LOCATION')}}"
      bucket: "{{kv('GCP_BUCKET_NAME')}}"
⚠️ Verify CSV filenames before running
The filenames above (job_postings.csv, job_skills.csv) are based on the known LinkedIn dataset structure. When you first download the dataset, check what files were actually created by looking at the Kestra execution logs after the download_dataset step (it logs ls -la *.csv). If the filenames differ, update the upload_*_to_gcs task from fields accordingly.

Data Warehouse — BigQuery Verification

Verify your raw tables are correctly structured. Understand why the partitioning and clustering from Phase 2 earns you full Data Warehouse rubric points (4/4).

💡 Why BigQuery specifically, not just Postgres?
BigQuery is an OLAP (Online Analytical Processing) columnar database built for analytics. When you ask "what's the average applicant count for entry-level jobs?", BigQuery only reads the two relevant columns across 1.3M rows instead of the entire table. Postgres (OLTP) reads entire rows. At this data scale, that difference translates to seconds vs minutes and pennies vs dollars in query costs. This is why data warehouses exist separately from operational databases.

💡 What dbt is actually doing
dbt lets you write plain SQL SELECT statements and turns them into tables or views in BigQuery automatically. Each .sql file in your models/ folder is one transformation. You just write the SELECT — dbt handles the CREATE TABLE AS part. It also manages build order (model B depends on model A → dbt runs A first), runs data quality tests, and generates a visual lineage diagram of your entire pipeline. This is the analytics engineering workflow you covered in Week 4.

Your dbt models in the repo are environment-agnostic SQL — they don’t care if they run against your BigQuery or someone else’s. The profile tells dbt where to connect. This separation is why someone can clone your repo, set up their own profile, and run dbt run against their own warehouse without touching your code. It directly supports your Reproducibility score.

💡 View vs Table — why stage as views and materialize marts as tables?
A view is a saved SQL query — no stored data, runs fresh every call. A table stores computed results. The strategy: staging and intermediate models are cheap views (they’re just cleaning/joining, accessed rarely). Mart models are real tables because your Looker Studio dashboard queries them repeatedly — a pre-materialized table responds instantly instead of recomputing on every dashboard load. This is standard dbt architecture used at companies like Airbnb, GitLab, and Spotify.

models/staging/stg_linkedin_jobs.sql 
-- Cleans and standardizes LinkedIn 2024 raw data.
-- Key jobs done here:
--   1. Rename columns to consistent names used across all models
--   2. Parse UNIX millisecond timestamps to real DATE type
--   3. Flag frontend roles by keyword-matching job titles
--   4. Normalize seniority labels (LinkedIn uses varied text like "Entry level", "Mid-Senior level")

WITH source AS (
  SELECT * FROM {{ source('frontend_jobs', 'raw_linkedin_jobs_2024') }}
),

cleaned AS (
  SELECT
    job_id,
    company_id,
    title                         AS job_title,
    formatted_experience_level    AS seniority_level,
    formatted_work_type           AS work_type,
    remote_allowed                AS is_remote,
    location,
    company_name,
    applies                       AS applicant_count,
    views                         AS listing_views,

    -- LinkedIn stores timestamps as UNIX milliseconds (not seconds)
    -- TIMESTAMP_MILLIS converts to proper TIMESTAMP, then DATE extracts the date
    DATE(TIMESTAMP_MILLIS(CAST(original_listed_time AS INT64))) AS posted_date,
    DATE(TIMESTAMP_MILLIS(CAST(expiry AS INT64)))               AS expiry_date,

    'linkedin_2024'               AS data_source,
    EXTRACT(YEAR FROM DATE(TIMESTAMP_MILLIS(CAST(original_listed_time AS INT64)))) AS post_year,

    -- Frontend role detection: TRUE if job title matches frontend keywords
    -- Adjust this list after previewing actual data titles from Phase 3
    CASE WHEN LOWER(title) LIKE ANY (
      '%frontend%', '%front-end%', '%front end%',
      '%react%', '%vue%', '%angular%',
      '%ui engineer%', '%ui developer%',
      '%web developer%', '%javascript developer%',
      '%next.js%', '%typescript developer%'
    ) THEN TRUE ELSE FALSE END    AS is_frontend_role,

    -- Normalize seniority labels to 3 clean categories
    -- LinkedIn uses labels like "Entry level", "Mid-Senior level", "Director" etc.
    CASE
      WHEN LOWER(formatted_experience_level) LIKE '%entry%'      THEN 'Entry Level'
      WHEN LOWER(formatted_experience_level) LIKE '%associate%'  THEN 'Entry Level'
      WHEN LOWER(formatted_experience_level) LIKE '%internship%' THEN 'Entry Level'
      WHEN LOWER(formatted_experience_level) LIKE '%mid%'        THEN 'Mid Level'
      WHEN LOWER(formatted_experience_level) LIKE '%senior%'     THEN 'Senior Level'
      WHEN LOWER(formatted_experience_level) LIKE '%director%'   THEN 'Senior Level'
      WHEN LOWER(formatted_experience_level) LIKE '%executive%'  THEN 'Senior Level'
      WHEN LOWER(formatted_experience_level) LIKE '%lead%'       THEN 'Senior Level'
      ELSE 'Unknown'
    END                           AS seniority_category

  FROM source
  WHERE title IS NOT NULL
    AND formatted_experience_level IS NOT NULL
)

SELECT * FROM cleaned

Intermediate Layer — Unify the Two Job Datasets

models/intermediate/int_jobs_unified.sql 
-- Combines LinkedIn 2024 and historical IT jobs into one unified table.
-- This is the single source of truth for all job-related mart models.
-- By the time data reaches here, both staging models have matching schemas
-- so UNION ALL works cleanly.

WITH linkedin AS (
  SELECT * FROM {{ ref('stg_linkedin_jobs') }}
),

historical AS (
  SELECT * FROM {{ ref('stg_it_jobs_historical') }}
),

unified AS (
  SELECT * FROM linkedin
  UNION ALL
  SELECT * FROM historical
)

SELECT
  *,
  -- Calculated field: how long ago was this posted?
  -- Used in ghost posting detection (mart_ghost_postings)
  DATE_DIFF(CURRENT_DATE(), posted_date, DAY) AS days_since_posted

FROM unified
WHERE posted_date IS NOT NULL
  AND post_year BETWEEN 2019 AND 2024
💡 Why {{ ref() }} instead of a table name?
{{ ref('stg_linkedin_jobs') }} is dbt’s way of referencing another model. dbt resolves it to the real BigQuery table name AND builds a dependency graph — it automatically knows to run stg_linkedin_jobs before int_jobs_unified. If you hard-coded a table name like `project.dataset.stg_linkedin_jobs`, dbt can’t see that dependency and might run models in the wrong order. The ref() function is what makes dbt pipelines reliable and self-documenting.

Mart Layer — Final Analytics Tables (One Per Research Question)

models/marts/mart_seniority_trends.sql — answers Question 1 
-- QUESTION 1: Has entry-level frontend share declined since 2019?
-- Aggregates frontend job postings by year and seniority category.
-- Calculates pct_of_total so you can plot a normalized trend line,
-- not just raw counts (which would be skewed by dataset size differences).

WITH jobs AS (
  SELECT * FROM {{ ref('int_jobs_unified') }}
  WHERE is_frontend_role = TRUE
    AND seniority_category != 'Unknown'
),

yearly_totals AS (
  SELECT post_year, COUNT(*) AS total_frontend_jobs
  FROM jobs GROUP BY 1
),

by_seniority AS (
  SELECT post_year, seniority_category, data_source, COUNT(*) AS job_count
  FROM jobs GROUP BY 1, 2, 3
)

SELECT
  s.post_year,
  s.seniority_category,
  s.data_source,
  s.job_count,
  t.total_frontend_jobs,
  ROUND(s.job_count / t.total_frontend_jobs * 100, 2) AS pct_of_total
FROM by_seniority s
JOIN yearly_totals t USING (post_year)
ORDER BY post_year, seniority_category
models/marts/mart_experience_inflation.sql — answers Question 2 
-- QUESTION 2: Are entry-level postings requiring senior-level skills?
-- "Experience inflation" = entry-level job title + demanding skill requirements.
-- Uses the LinkedIn skills table joined back to postings by job_id.

WITH entry_jobs AS (
  SELECT * FROM {{ ref('stg_linkedin_jobs') }}
  WHERE is_frontend_role = TRUE
    AND seniority_category = 'Entry Level'
),

job_skills AS (
  SELECT * FROM {{ source('frontend_jobs', 'raw_linkedin_skills') }}
),

skills_analysis AS (
  SELECT
    j.job_id,
    j.job_title,
    j.company_name,
    j.posted_date,
    j.applicant_count,
    j.location,
    COUNT(s.skill_abr)           AS total_skill_count,

    -- Count skills that signal senior/complex work in an entry-level posting
    COUNTIF(LOWER(s.skill_abr) IN (
      'system design', 'aws', 'gcp', 'azure', 'kubernetes',
      'docker', 'ci/cd', 'microservices', 'architecture',
      'leadership', 'terraform', 'kafka'
    ))                           AS senior_skill_count,

    -- Inflation score: what % of required skills are senior-level?
    -- High score = entry job that expects senior knowledge
    ROUND(
      COUNTIF(LOWER(s.skill_abr) IN (
        'system design', 'aws', 'gcp', 'azure', 'kubernetes',
        'docker', 'ci/cd', 'microservices', 'architecture',
        'leadership', 'terraform', 'kafka'
      )) / NULLIF(COUNT(s.skill_abr), 0) * 100, 2
    )                            AS inflation_score

  FROM entry_jobs j
  LEFT JOIN job_skills s USING (job_id)
  GROUP BY 1,2,3,4,5,6
)

SELECT * FROM skills_analysis
ORDER BY inflation_score DESC NULLS LAST
models/marts/mart_ghost_postings.sql — answers Question 3 
-- QUESTION 3: Which companies show ghost posting patterns?
-- Ghost posting proxy: high applicant count + long listing duration.
-- Uses LinkedIn 2024 only (historical dataset lacks applicant counts).
-- Threshold: 100+ applicants AND 60+ days since posted = potential ghost job.

WITH linkedin_frontend AS (
  SELECT * FROM {{ ref('int_jobs_unified') }}
  WHERE data_source = 'linkedin_2024'
    AND is_frontend_role = TRUE
    AND applicant_count IS NOT NULL
),

flagged AS (
  SELECT
    job_id,
    job_title,
    company_name,
    seniority_category,
    applicant_count,
    posted_date,
    days_since_posted,
    location,
    is_remote,

    CASE
      WHEN applicant_count >= 100
       AND days_since_posted >= 60 THEN TRUE
      ELSE FALSE
    END AS is_potential_ghost

  FROM linkedin_frontend
),

-- Aggregate to company level for the dashboard bar chart
company_summary AS (
  SELECT
    company_name,
    COUNT(*)                          AS total_frontend_postings,
    COUNTIF(is_potential_ghost)       AS ghost_posting_count,
    ROUND(COUNTIF(is_potential_ghost) / NULLIF(COUNT(*), 0) * 100, 1) AS ghost_pct,
    AVG(applicant_count)              AS avg_applicants,
    AVG(days_since_posted)            AS avg_days_listed
  FROM flagged
  GROUP BY 1
  HAVING total_frontend_postings >= 3   -- exclude companies with only 1-2 posts
)

SELECT * FROM company_summary
ORDER BY ghost_pct DESC

# Frontend Job Market Analysis Pipeline

## Problem Statement
The frontend engineering job market has experienced significant shifts in recent years,
with debates around AI impact and a perceived decline in entry-level opportunities.
This project builds an end-to-end batch data pipeline analyzing job postings from
2019-2024 -- examining seniority trends, experience inflation, and ghost posting
patterns -- to surface data-driven insights into who's actually hiring frontend
engineers and what they actually expect.

## Questions Answered
1. Has the share of entry-level frontend roles declined since 2019?
2. Are entry-level postings experiencing "experience inflation" (junior titles + senior skills)?
3. Which companies show ghost posting patterns (high applicants, long-listed, never filled)?

## BigQuery Optimization
Tables are **partitioned** by posting date and **clustered** by seniority level.

- **Partitioning** by `DATE(original_listed_time)` means queries filtered to a date range
  (e.g., "show 2024 data only") scan only the relevant date partitions instead of the full
  1.3M row table — dramatically reducing query time and cost.
- **Clustering** by `formatted_experience_level` sorts rows within each partition so
  seniority-filtered queries (e.g., "Entry level only") skip irrelevant rows entirely.

## How to Reproduce

### Prerequisites
- GCP account with billing enabled
- [Terraform](https://developer.hashicorp.com/terraform/install) installed
- [Docker Desktop](https://www.docker.com/products/docker-desktop/) installed
- Python 3.10+ with pip
- [Kaggle account](https://www.kaggle.com) and API token at `~/.kaggle/kaggle.json`

### Steps
1. Clone this repository
2. Create a GCP service account with Storage Admin + BigQuery Admin roles, download JSON key
3. Store key at `~/.gcp/fe-jobs-sa.json`
4. Copy `terraform/terraform.tfvars.example` to `terraform/terraform.tfvars`, fill in your values
5. `cd terraform && terraform init && terraform apply`
6. `cd kestra && docker compose up -d`
7. In Kestra UI (localhost:8080): run flows in order: 01 → 02 → 03 → 04
8. `pip install dbt-bigquery`
9. Configure `~/.dbt/profiles.yml` (see `dbt/profiles.example.yml`)
10. `cd dbt/frontend_jobs_dbt && dbt run`
11. Connect Looker Studio to BigQuery marts (see Phase 5 of project guide)
